# Bound Context

Read-only critique found that `_get_from_yaml()` returns `{}` / `[]` for column `meta` / `tags` when a YAML column exists but lacks those fields, suppressing manifest fallback. It also noted the new inheritance test directly writes `column.config.*`, which may fail on older dbt rows if a column object lacks `config`.

# Parent Launch Checklist

# Source Snapshot

Current uncommitted `_get_from_yaml()` uses `_get_effective_column_meta()` and `_get_effective_column_tags()` for YAML column dicts. Those helpers return empty containers when no effective value exists. `PropertyAccessor.get(source="yaml")` only falls back to manifest when `_get_from_yaml()` returns `None`.

# Change Class

# Verification Targets

- ticket:c10meta02#ACC-004
