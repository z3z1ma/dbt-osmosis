# Mission

Serialize YAML sync work by resolved target YAML path so no two parallel sync workers can mutate or write the same ruamel document, while preserving independent-file concurrency and existing versioned-model behavior.

# Bound Context

`ticket:c10race13` is high-risk `code-behavior` work under `initiative:dbt-110-111-hardening#OBJ-003`. YAML preservation is a core correctness boundary: schema sync must not silently lose model/source entries, user-authored sections, or versioned entries because multiple workers touched the same YAML file.

Stay inside sync scheduling, YAML write temp-file safety, and focused regression coverage. Do not rewrite the transform pipeline, schema parser/reader architecture, or cache internals unless the red test proves grouping alone cannot satisfy the ticket.

# Source Snapshot

Relevant current implementation state:

- `src/dbt_osmosis/core/sync_operations.py::_sync_single_node_to_yaml()` resolves a node's current path, falls back to `get_target_yaml_path()` when needed, prepares one YAML doc, mutates the node section, and finalizes the document.
- `_group_sync_nodes()` currently groups only versioned models by `(package_name, name, patch_path/original_file_path/unique_id)`. Non-versioned models and sources always become singleton groups.
- `sync_node_to_yaml(context, node=None)` calls `context.pool.map()` over `_group_sync_nodes(context)`, so singleton groups for different nodes can run concurrently even when the nodes route to the same YAML path.
- `_read_yaml()` protects cache reads/writes with `_YAML_BUFFER_CACHE_LOCK`, but it returns a shared mutable document after the lock is released; path grouping should prevent two workers from mutating the same returned object concurrently.
- `schema/writer.py::_write_yaml()` and `commit_yamls()` currently derive temp files with `path.with_suffix(path.suffix + ".tmp")`. If concurrent writers ever target the same file, or if multiple processes share a target path, that fixed temp name can collide. Use a unique temp path in the same directory while preserving atomic replace and cleanup behavior.
- Existing regression coverage in `tests/core/test_sync_operations.py::test_sync_node_to_yaml_all_versions_share_one_truthful_write` covers the versioned model grouping bug, but there is no path-based grouping regression for unversioned models or sources sharing a YAML file.

# Change Class

`code-behavior`, high risk. The implementation affects concurrency and YAML data preservation, so prefer the smallest change that makes same-target work sequential without removing concurrency for independent target files.
