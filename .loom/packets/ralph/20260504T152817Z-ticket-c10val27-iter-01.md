# Mission

Make model validation timeout semantics truthful: a validation execution that exceeds `timeout_seconds` must return `ModelValidationStatus.TIMEOUT` promptly in tests, while compile and execution error classification remains unchanged.

# Bound Context

`ticket:c10val27` is medium-risk `code-behavior` work under `initiative:dbt-110-111-hardening#OBJ-006`. The current code exposes `timeout_seconds` but only classifies timeout by looking for the word `timeout` in an exception string after `execute_sql_code()` returns or raises.

The ticket allows either real timeout enforcement or honest API/docs changes if true cancellation cannot be guaranteed. For this iteration, prefer the smallest implementation that makes the public validation function return promptly for a timed-out execution without claiming adapter-level cancellation that cannot be guaranteed.

# Source Snapshot

Current implementation reality at packet compilation:

- `src/dbt_osmosis/core/validation.py::validate_models()` accepts `timeout_seconds` and passes it into `_validate_single_model()`.
- `_validate_single_model()` compiles SQL, calls `execute_sql_code(context, compiled_sql)` synchronously, measures elapsed time, and returns success.
- Timeout is only recognized in the `except` block when `timeout_seconds` is set and the exception message contains `timeout`.
- `tests/core/test_model_validation.py` covers success, compile error, and execution error with mocked `compile_sql_code()` and `execute_sql_code()`, but it has no timeout test.

# Change Class

`code-behavior`, medium risk. The change affects validation runtime behavior and can affect CI/user workflows that rely on validation not hanging indefinitely.

Implementation summary:

- Added a prompt timeout path around `execute_sql_code()` in `core/validation.py`.
- When `timeout_seconds` is `None` or `<= 0`, execution remains synchronous to preserve existing behavior.
- When `timeout_seconds` is positive, validation runs `execute_sql_code()` in a daemon `threading.Thread`, waits with `thread.join(timeout=timeout_seconds)`, and raises an internal `_ModelValidationTimeoutError` if the worker is still alive.
- `_validate_single_model()` catches that internal timeout and returns `ModelValidationStatus.TIMEOUT` with the compiled SQL, elapsed wait time, and an explicit message that dbt-osmosis stopped waiting but did not cancel the underlying adapter query.
- Existing compile-error and execution-error classification remains intact, including the previous string-based timeout classification for exceptions raised by the adapter path.
- Added `test_validate_model_execution_timeout_returns_promptly`, which mocks a slow execution with `time.sleep(0.2)`, requests a `0.01s` timeout, and asserts the call returns promptly with `TIMEOUT`, compiled SQL, bounded execution time, and a timeout error message.

Cancellation/cleanup limitations:

- This implementation stops waiting and returns a timeout result; it does not cancel a running warehouse/adapter query.
- The daemon worker thread may continue until the adapter call returns and may hold adapter/context resources during that period.
- Repeated real adapter timeouts could temporarily accumulate daemon worker threads/connections until their underlying calls finish.
