# Source Snapshot

Current implementation reality:

- `src/dbt_osmosis/core/config.py:24-25` imports `ModelParser` from `dbt.parser.models`.
- `src/dbt_osmosis/core/config.py:504-547` calls `ModelParser.parse_from_dict(None, node)` inside `_add_cross_project_references()` for exposed dbt-loom model nodes.
- `src/dbt_osmosis/core/config.py:430-438` defines a `DbtProjectContext.manifest` setter that writes `_DbtProject__manifest` with `object.__setattr__`, but the private mutation is not isolated behind a named compatibility shim.
- `src/dbt_osmosis/core/config.py:624-640` catches dbt-loom import/addition failures so projects without dbt-loom or with broken loom manifests do not fail project context creation.
- `tests/core/test_config.py` currently has no focused `_add_cross_project_references()` regression test.

Research snapshot:

- `research:dbt-110-111-api-surfaces` observed dbt-core v1.10.0 and v1.11.0 define `ModelParser.parse_from_dict(self, dct, validate=True)`, so passing `None` as `self` is private/brittle.
- The ticket suggests `ModelNode.from_dict(...)` as the likely valid path if it accepts dbt-loom node dictionaries.
