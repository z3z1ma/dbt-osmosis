# Mission

Implement the supported policy for cross-section YAML anchors and aliases in schema read/write cycles: aliases between unmanaged top-level sections and managed model/source sections must remain valid and parseable after dbt-osmosis writes managed changes.

# Bound Context

`ticket:c10anchors30` advances `initiative:dbt-110-111-hardening#OBJ-003`: schema YAML preservation should not lose or corrupt user-authored dbt content during mutation.

The ticket is bounded to these claims:

- `ticket:c10anchors30#ACC-001`: explicit policy for cross-section YAML anchors and aliases.
- `ticket:c10anchors30#ACC-002`: regression coverage for `x-common-tests: &common` and `data_tests: *common`.
- `ticket:c10anchors30#ACC-003`: supported write/read round-trip keeps aliases valid and parseable by ruamel/dbt-compatible YAML parsing.
- `ticket:c10anchors30#ACC-005`: preserved unmanaged sections remain preserved under the support policy.

`ticket:c10anchors30#ACC-004` is intentionally not the chosen route because parent characterization found a practical support path rather than requiring an explicit unsupported failure mode.

# Source Snapshot

Relevant current observations from parent inspection at source fingerprint `2c3cf96e102360a2e09af6a4e4d9d4a892956276`:

- `src/dbt_osmosis/core/schema/reader.py:210-229` currently reads each schema file twice: once through filtered `OsmosisYAML.load()` and once through an unfiltered `ruamel.yaml.YAML()` instance. Cross-section alias relationships are split into two separate ruamel object graphs before writer merge.
- `src/dbt_osmosis/core/schema/parser.py:25-41` partitions top-level keys into managed and preserved sections. The partition copies top-level values, so a single parsed object graph can preserve alias identity between managed and unmanaged sections.
- `src/dbt_osmosis/core/schema/writer.py:35-63` starts merged output from filtered managed data and appends preserved sections afterward. With a shared object graph this keeps aliases valid but can move the anchor definition from the unmanaged top-level key into the managed section when the managed alias appears first in output order.
- Parent characterization command using `_read_yaml()` and `_write_yaml()` on a file with `x-common-tests: &common_tests` and model column `data_tests: *common_tests` produced parseable YAML, but the written file expanded `data_tests` into a sequence and wrote `x-common-tests` as a second full sequence. The alias relationship was not preserved.
- Parent characterization of a single unfiltered parse plus top-level partition preserved the alias relationship when the writer dumped the shared object graph, especially when merge output followed original top-level key order.
- Existing tests in `tests/core/test_schema.py:249-327` cover filtering/preserving `anchors` and unknown top-level sections, but not cross-section aliases between unmanaged and managed sections.
- `src/dbt_osmosis/core/AGENTS.md` applies to `core/` edits: keep ruamel.yaml, avoid broad refactors, preserve parser/reader/writer boundaries, and respect cache discipline.
