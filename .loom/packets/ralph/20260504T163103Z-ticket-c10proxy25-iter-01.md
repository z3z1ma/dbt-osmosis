# Mission

Make the SQL proxy support boundary explicit and safe: keep it experimental opt-in, keep proxy-only dependencies out of base installs, avoid lossy SQL reserialization, and document/test that comment middleware is in-memory only.

# Bound Context

`ticket:c10proxy25` is medium-risk `code-behavior` work under `initiative:dbt-110-111-hardening#OBJ-007` and `#OBJ-008`. Human input is required only if the proxy is removed or promoted from experimental to supported. This iteration should not remove the proxy or promote it to supported; preserve it as experimental opt-in.

# Source Snapshot

Current state at packet compilation:

- `pyproject.toml` has `sqlglot` in base dependencies because SQL linting uses it, and `mysql-mimic` in the `proxy` extra only.
- `tests/test_package_metadata.py` already checks `mysql-mimic` is not a base dependency, the `proxy` extra exists, and docs state the extra is dependency-only.
- `src/dbt_osmosis/sql/proxy.py` imports `mysql_mimic` at module import time, executes SQL using `expression.sql(dialect=self.project.adapter.type())`, and has comment middleware that mutates only the in-memory manifest while the docstring says future YAML writeback could happen.
- No current tests directly cover proxy SQL preservation or comment middleware behavior.

# Change Class
