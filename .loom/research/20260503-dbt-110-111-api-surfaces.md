# Question

Which dbt 1.10.x and 1.11.x API and artifact surfaces touched by `dbt-osmosis` create concrete compatibility risk?

# Why This Matters

`dbt-osmosis` declares `dbt-core>=1.8` and imports internal dbt parser/task APIs while also reading/writing manifest-backed column and config metadata. dbt 1.10 introduced important `config.meta` and `config.tags` placement changes for columns, and dbt 1.11 continues those shapes. The project needs a durable, citable synthesis for tickets that target 1.10.x/1.11.x compatibility.

# Scope

Covered:

- dbt docs for `meta`, column properties, and deprecations.
- dbt-core v1.10.0 and v1.11.0 source for `ColumnConfig`, `ColumnInfo`, `ModelParser`, `SqlBlockParser`, and `SqlCompileRunner`.
- Local `dbt-osmosis` code paths that read/write manifest columns, resolve settings, parse dbt-loom nodes, and compile SQL.

Excluded:

- Full runtime verification across every dbt patch and adapter.
- dbt Fusion behavior beyond noting that schema-version-only detection is too broad.
- A final support policy for dbt versions beyond 1.11.x.

# Method

The investigation combined five read-only oracle audits, targeted local source reads, dbt docs fetches, GitHub code search, and raw source fetches from dbt-core v1.10.0 and v1.11.0.

# Evidence Synthesis

dbt docs state that column-level `meta` moved under `config.meta` in v1.10 and was backported to 1.9. The `columns` reference shows column `config.tags` and `config.meta` as supported sub-properties. The deprecations page warns that custom configs and column top-level `meta` should move under `config.meta`, and that custom top-level config keys are reserved in dbt Core v1.10 and Fusion.

dbt-core v1.10.0 and v1.11.0 both define `ColumnConfig` with `meta: Dict[str, Any]` and `tags: List[str]`. Both versions define `ColumnInfo.config: ColumnConfig = field(default_factory=ColumnConfig)`, while also retaining legacy `ColumnInfo.meta` and `ColumnInfo.tags` fields.

Local code in `src/dbt_osmosis/core/transforms.py:356-370` creates `ColumnInfo.from_dict(...)` for injected missing columns and then deletes the `config` attribute. Because `ColumnInfo.config` is part of the dbt dataclass in both audited versions, deleting it creates an invalid object for dbt serialization paths such as `to_dict`.

Local settings/property/inheritance code still reads column-level options primarily from `column.meta` and `column.tags`. This is incomplete for dbt 1.10/1.11 users who place column settings under `columns[].config.meta` and `columns[].config.tags` according to current docs.

dbt-core v1.10.0 and v1.11.0 both define `ModelParser.parse_from_dict(self, dct, validate=True) -> ModelNode`. Local code calls `ModelParser.parse_from_dict(None, node)` in `src/dbt_osmosis/core/config.py:529`, which relies on passing `None` for `self` into an instance method. The current method body happens to call only class/static operations, but this is not a stable public API usage.

dbt-core v1.10.0 and v1.11.0 both define `SqlBlockParser.parse_remote(self, sql: str, name: str) -> SqlNode` and `SqlCompileRunner.compile(self, manifest: Manifest)`. Local SQL helpers rely on these internal APIs. The signatures matched in the audited tags, but the private nature of the APIs means the compatibility guarantee should come from real matrix tests and an isolated shim, not from broad package constraints alone.

# Rejected Options

- Rely only on dbt docs for compatibility. Rejected because local code imports private dbt parser/task APIs where docs do not define a stable contract.
- Treat manifest schema versions greater than v12 as proof of dbt Fusion. Rejected because future dbt-core schema versions could exceed v12 without being Fusion.
- Preserve compatibility by deleting `ColumnInfo.config` to avoid empty YAML output. Rejected because it mutates dbt objects into a shape that conflicts with dbt-core v1.10/v1.11 dataclass fields.

# Null Results

No runtime dbt 1.10.x/1.11.x matrix was run during this research pass. The investigation is source/docs/audit evidence, not behavioral validation.

# Conclusions

- dbt 1.10.x and 1.11.x require first-class support for `ColumnInfo.config.meta` and `ColumnInfo.config.tags`; legacy top-level `meta`/`tags` cannot be the only read source.
- Deleting `ColumnInfo.config` is a concrete bug under both dbt 1.10.0 and 1.11.0 source shapes.
- `ModelParser.parse_from_dict(None, node)` is a brittle internal API usage that should be replaced by `ModelNode.from_dict(...)` or another valid parse path.
- SQL compile/run internals are not immediately signature-broken in v1.10.0/v1.11.0, but they need compatibility tests and isolation because they rely on private dbt APIs.
- CI and dependency policy should not claim broad dbt 1.11 support until matrix coverage asserts the actual installed core/adapter versions and runs meaningful fixture behavior.

# Recommendations

- Prioritize ticket:c10col01, ticket:c10meta02, ticket:c10ci06, ticket:c10fix11, ticket:c10cfg12, and ticket:c10sql21 before claiming dbt 1.10/1.11 compatibility.
- Use current dbt docs as the expected user-facing config shape, but verify actual manifest fields from parsed fixtures under dbt 1.10.x and 1.11.x.
- Add an internal compatibility shim or sharply scoped module boundary for private dbt parser/task usage.
- Narrow unsupported future dbt versions or add scheduled canary coverage before allowing unreviewed dbt minors through package resolution.
