# Objective

Make `dbt-osmosis` confidently compatible with dbt Core 1.10.x and 1.11.x while hardening the build, CI, fixtures, packaging, and high-friction code paths that make compatibility work unreliable.

# Why Now

The current package constraint allows `dbt-core>=1.8` with no upper bound, but the code imports several dbt internal APIs and only partially understands dbt 1.10+ config shapes. The current CI matrix treats dbt 1.11 as a narrow canary rather than a support guarantee, and release/build/docs workflows can rot or publish before validation.

# In Scope

- dbt 1.10.x and 1.11.x manifest, parser, config, SQL task, and column metadata compatibility.
- YAML sync correctness, cache isolation, source/model/versioned model handling, and preservation behavior.
- CLI behavior for YAML, SQL, generate, lint, diff, test suggestions, and workbench command families.
- Workbench optional dependency boundaries, target switching, Streamlit invocation, and external feed safety.
- CI, release, docs build, dependency resolution, packaging extras, dev tooling, fixture isolation, and version assertions.
- Cleanup tickets for dead code, private-state mutation, leaky abstractions, YAGNI surfaces, and experimental public-ish code.

# Out Of Scope

- Implementing every ticket inside this initiative in the same backlog-creation pass.
- Dropping existing supported dbt/Python versions without an explicit compatibility decision and migration guidance.
- Treating dbt Fusion compatibility as guaranteed unless a ticket explicitly proves that behavior.
- Large rewrites that break public CLI behavior, YAML output contracts, package APIs, or documented workflows without a deliberate migration plan.

# Delegated Authority / Autonomy Boundaries

Agents may triage, research, file, sequence, and implement bounded tickets under this initiative without asking when the work preserves public behavior and stays inside an individual ticket's scope.

Human decision triggers:

- Removing or narrowing advertised dbt, adapter, Python, CLI, or package support.
- Removing the SQL proxy or changing it from experimental to supported.
- Introducing breaking YAML output changes or public CLI flag changes.
- Publishing, tagging, or releasing packages.

# Objective-Level Stop Conditions

Stop and return to planning or the user if dbt 1.11 adapter availability prevents a meaningful support matrix, if a compatibility fix implies a breaking public behavior change, if a ticket uncovers security-sensitive secret exposure, or if a core dbt API surface lacks a stable replacement and requires a project-level support decision.

# Success Metrics

- OBJ-001: dbt 1.10.x and dbt 1.11.x compatibility is exercised by CI with explicit installed-version assertions and meaningful adapter-backed smoke/integration coverage.
- OBJ-002: dbt 1.10+ `config.meta` and `config.tags` shapes are supported for settings resolution, property access, inheritance, and YAML output without regressing legacy top-level `meta`/`tags` inputs.
- OBJ-003: YAML sync is deterministic, preserves non-osmosis-owned content, avoids concurrent writes to the same target document, and does not silently delete user-authored YAML sections.
- OBJ-004: Tests and integration fixtures do not depend on source-tree generated dbt artifacts, repo-root DuckDB files, stale manifests, or cache replacement that masks production behavior.
- OBJ-005: Release, package, docs, and dependency workflows validate artifacts before irreversible tags or publishes and use reproducible dependency resolution.
- OBJ-006: User-facing CLI, SQL, workbench, generate, lint, diff, and optional AI paths fail clearly, respect documented selectors/options, and have coverage for the behavior they advertise.
- OBJ-007: Experimental or leaky surfaces are either made intentionally supported with tests/docs/dependencies or isolated from base installs and public expectations.
- OBJ-008: Trash code and YAGNI cleanup removes private-state mutation, dead stubs, duplicated dependency sources, and misleading abstractions after higher-risk compatibility issues are addressed.

# Milestones

- M1: Compatibility evidence and CI baseline are made trustworthy for dbt 1.10.x and 1.11.x.
- M2: Core YAML/config/inheritance bugs that threaten correctness or data preservation are fixed and covered.
- M3: CLI, SQL, workbench, generate, lint, diff, LLM, and proxy surfaces are reconciled with actual behavior and optional dependency boundaries.
- M4: Build, release, docs, packaging, dependency, and dev-tooling surfaces are deterministic enough to support releases.
- M5: Cleanup and YAGNI tickets remove misleading or stale code after functional compatibility work lands.

# Dependencies

- dbt Core and adapter availability for the claimed support matrix, especially dbt-duckdb compatibility with dbt 1.11.x.
- `uv`, GitHub Actions, PyPI publishing, Docusaurus, and Node dependency ecosystems.
- Existing public CLI/YAML behavior and downstream consumers described in `constitution:main`.

# Risks

- dbt internals can change under private imports, so compatibility fixes should prefer shims, real integration tests, and explicit version assertions.
- Adding a full dbt/Python matrix can become slow or flaky unless fixture isolation and dependency resolution are fixed first.
- YAML preservation bugs can cause silent user data loss, which is higher risk than visible command failure.
- Optional AI/workbench/proxy paths can drag large or incompatible dependencies into base installs if extras are not rationalized.
